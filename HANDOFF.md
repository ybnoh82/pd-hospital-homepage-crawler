# HANDOFF — 비용·품질 최적화 진행 노트

다음 세션이 이어받기 위한 인수인계 문서. 운영 가이드·아키텍처는 `CLAUDE.md`에 있고,

## 0. 한 줄 현황

목표는 **병원당 평균 $1·5분 이내로 8,000개를 신뢰성 있게 크롤링**. 비용·시간·출력유효성·**데이터
무손실**(§3·§4-1)은 구조적으로 해결됨(캡 + 러너 repair + 시술→제품/장비 자동집계). 남은 핵심 과제는
**비타깃·죽은 사이트에 비용 안 쓰기 = prefetch 트리아지**(§6-1, 실제 8,000 배치에서만 의미 — 큐레이션된
sample10엔 빈 사이트가 없어 검증 불가). opus/high·$3·900s·10병렬 sample10 실측: 합계 $28.71, 누수 0,
에이전트 매칭 품질 양호(unmatched의 낮은 매칭률은 정당한 보류 — 결정론적 매칭 패스는 거짓매칭 양산, 금지).

  - 시간캡 78%에서 "지금 유효 JSON 저장하고 종료" **소프트 데드라인 steering** 주입.

    실행 전 기존 출력 삭제(오염 방지), `ResultMessage` 비용 백필.
  - CLI: `--budget --model --effort --time-limit --from-csv --id --name ...`
- **`SKILL.md`** — §0-6에 **첫 체크포인트 저장**(정찰 직후·매칭 전·`sample_output.json` 골격) 추가.
  "작업 예산" 섹션에 증분 저장 규칙. §8은 최종 덮어쓰기.
- **`output_scheme.py`** — `--repair <path> --id --name --url` 모드 추가. 유효 스켈레톤 위에
  에이전트 출력을 얹고 무효 위치만 떨궈 항상 스키마 통과. 잘 쓴 파일엔 no-op. USEFUL/EMPTY 판정.
- **`data/sample10.csv`** — 원본 컬럼 + `homepage_url`. 사이트 형태 다양한 10개(§5).
- **미구현(다음 할 일)**: 배치 드라이버, prefetch 분류기 — §6.

## 2. 핵심 실측 (반복하며 측정한 것 — 다시 측정하지 말 것)

**모델 (가장 큰 레버):**
- Haiku 4.5($1/$5): 가장 싸지만 이 멀티스텝 에이전트(SPA·매칭·vision)에서 **조기 종료·미저장으로 완주 실패**. 부적합.
- Sonnet 4.6($3/$15): 안정적 완주. **기본 모델로 확정.** Opus 4.8($5/$25)는 하드 사이트용.
- effort low↔medium 비용차는 캡에 가려 작음. low가 빨라 기본값.

**비용·시간 (강남 5개 실사이트, 무캡 자연완주):**
- mimo21 4.4분/$1.03, faceup 5.9분/$1.02, toxnfill 8.1분/$1.15, uni114 9.2분/$1.29, lifton 5.2분/$1.02 → **평균 6.5분/$1.10**.
- 교훈: **미용·성형 사이트는 전부 콘텐츠가 풍부해 항상 캡에 닿는다.** "쉬운 사이트가 평균을 낮춘다"는 가정은 **틀렸다**(실측).
- **$ 예산은 시간을 못 묶는다** — 느린 순차 왕복 사이트는 비용 적게 쓰며 9분까지 흐름. 그래서 **시간캡(wall-clock)** 별도 필요.

**$2·10분 품질 실측(최근):** 유앤아이 제품13m/장비13m/시술18/의료진12(최상), 페이스업 4+2/6+1/23/1,
리프톤 9+3/장비0+8/24/7, 권장덕 0/0/25/1(수술중심—제품 원래 적음), **톡스앤필 0/0/61/0(품질 불량—§4)**.
캡 올리니 **의료진·매칭이 살아남**(타이트 캡이 품질 킬러였음). 비용 $1.33~$2.02, 다수 $2 캡 도달.

**prefetch(curl·무브라우저·무LLM) 실측:** raw HTML만으로 사이트 특징 상당 부분 파악 가능 —
리프톤=SPA쉘(2KB·글자8·키워드0→브라우저필수), 유앤아이=텍스트풍부(44K·키워드12/12→무브라우저 추출가능),
mimo21=이미지형(글자적음·img37·키워드0→vision필요), 미인=WordPress(빌더감지). **한계: JS렌더 내용은 안 보임.**

**머신/병렬:** 24GB·12코어. 한 실행 ~1GB(claude CLI 390MB+playwright 300MB+chromium~400MB+python). **동시 8-12개 안전.**
CPU 아닌 메모리 바인딩. **8,000 배치의 진짜 천장은 머신이 아니라 API 레이트리밋(TPM/RPM).**
현재 Claude Code 로그인(구독) 인증 사용 — 지속 배치엔 API 키+tier 필요.

## 3. 신뢰성 해결 방식 (중요 — 에이전트에 의존하지 않음)

이전엔 "유효 저장"이 에이전트 신뢰도에 의존(~70%). 이제 **러너가 결정론적으로 100% 유효 보장**:
실행 후 `--repair`가 (1) 잘 쓴 파일은 그대로, (2) 무효 파일은 유효 부분만 살리고 무효 부분 제거,
(3) 미저장은 스켈레톤 생성. 실측: 계속 실패하던 uni114·toxnfill가 모두 유효 파일로(uni114 미매칭 제품19·장비23 보존).
**데이터 충분성은 USEFUL/EMPTY(exit 1)로 플래그** → 배치가 EMPTY만 재시도.

**[2026-06-16] 데이터 무손실 보장 추가** — repair가 무효 항목을 *삭제*하는 탓에, 에이전트가 제품·장비를
틀린 필드명으로 쓰면 1순위 가치 출력이 조용히 증발하는 구멍이 있었다(§4-1 톡스앤필). 이제 `--repair`가
`aggregate_from_treatments()`로 **시술(treatments)에 박힌 제품·장비 이름을 최상위 unmatched로 자동 집계**한다
(이미 matched/unmatched에 있으면 no-op, 카탈로그 매칭은 안 함). "거둔 건 절대 안 흘린다"를 러너가 보장.

## 4. 미해결 이슈

1. ~~**톡스앤필 추출 누락**~~ **[해결 2026-06-16]** — "제품·장비 0, 시술만 61"은 빈 사이트·구조
   문제가 아니라 **러너의 데이터 손실 버그**였다. 크롤은 잘 됐고(시술에 제품17·장비13이
   `treatments[].product_name/equipment_name`에 박혀 있었음) 최상위 `products`/`equipments`로
   집계되지 못해 사라진 것. `--repair`가 스키마 필수 필드 없는 `matched_*`/`unmatched_*`를
   **통째로 삭제**하므로, 에이전트가 틀린 필드명으로 쓰면 1순위 가치 출력이 조용히 증발하고
   USEFUL(exit0)로 통과해 재시도도 안 됐다. **수정**: `aggregate_from_treatments()`(§3)로 시술의
   제품·장비 이름을 unmatched로 자동 집계 → "거둔 건 안 흘림"을 결정론적 보장. opus/high sample10
   재실행에서 톡스앤필 제품18+13·장비7+5, 10개 전부 누수 0 검증.
2. **시간캡 실행은 비용 미기록** — 타임캡으로 끊기면 `ResultMessage`가 안 와 비용 백필 안 됨(유앤아이 cost=None).
   배치 비용 집계를 위해 경과시간 기반 추정 또는 다른 비용 소스 필요.
3. **리프톤 장비 매칭 미완** — 장비 8개 다 unmatched(매칭을 시간 안에 못 끝냄). effort medium이 도움될 수 있음.
4. **완전성 vs $1·5분 근본 긴장** — 콘텐츠 풍부 사이트는 {$1,5분,완전데이터} 동시 불가. 현재는 $1·5분·유효성을
   하드 보장하고 완전성만 캡으로 트레이드. 품질을 원하면 $2/10분처럼 캡을 올려야 함(사용자 현재 선택).

## 7. 재개 방법 (명령어)

```bash
# 한 병원 (기본 캡: $0.90/270s)
uv run python main.py <URL> --from-csv data/beauty_hospitals_gangnam.csv --id <ID>
# 품질 우선 ($2/10분)
uv run python main.py <URL> --from-csv data/sample10.csv --id <ID> --budget 2 --time-limit 600
# 5개 병렬 (예시)
for id in smado8 6wzs3t e2atn2 kgeuid pja1wj; do
  url=$(uv run python -c "import csv;print({r['id']:r['homepage_url'] for r in csv.DictReader(open('data/sample10.csv'))}['$id'])")
  uv run python -u main.py "$url" --from-csv data/sample10.csv --id "$id" --budget 2 --time-limit 600 > /tmp/$id.log 2>&1 &
done
# 결과 검증/복구(단독)
uv run python .claude/skills/hospital-homepage-extract/reference/output_scheme.py output/<file>.json   # 검증
# 빠른 사이트 사전분류(브라우저·LLM 없이)
curl -sL -A "Mozilla/5.0" --max-time 20 <URL> | wc -c   # SPA면 작음
```

결과는 `output/{id}_{이름}_homepage.json`. 러너 요약줄에 `유효·데이터있음`/`유효·빈스켈레톤(재시도권장)` 표시.

## 8. `/goal` 스킬 활용 가이드

`/goal <조건>` 은 **세션 범위 Stop 훅**을 건다 — 그 조건이 충족됐다고 판단될 때까지 에이전트가
멈추지 못하고 계속 작업·측정한다. 이번 세션의 "$1·5분·8,000개" 목표가 그 예였다.

- **언제 쓰나**: 측정 가능한 목표로 반복 최적화를 강제하고 싶을 때.
- **잘 쓰는 법**: 조건을 **검증 가능·구체적**으로. 모호하면("품질 좋게") 에이전트가 종료 판단을 못 해 계속 돈다.
  - 좋은 예: "sample10.csv 10개를 돌려 유효율 100%·평균 $1.5 이하·평균 7분 이하를 실측으로 보여라."
  - 나쁜 예: "데이터 품질을 높여라"(측정 기준 없음).
- **비용 주의**: 목표 달성을 위해 에이전트가 실제 크롤링을 반복 실행하면 $가 든다(이번 세션도 다수 $1~2 실행 소모). 조건에 **비용·실행횟수 상한**을 함께 넣는 게 안전.
- **해제**: 조건 충족 시 자동 해제. 일찍 끝내려면 `/goal clear`.
- **다음 세션 권장 목표 예시**:
  `/goal data/sample10.csv 10개를 prefetch 분류 후 적정 경로로 돌려, 유효율 100%·EMPTY 2개 이하·평균비용 $1.5 이하를 표로 실측 제시. 총 실행 비용 $25 넘기지 말 것.`
