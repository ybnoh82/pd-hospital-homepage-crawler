# 병원 홈페이지 추출

공식 홈페이지 URL을 받아 정규화 JSON을 만든다. **데이터 출처는 주어진 홈페이지 하나다** — 그 사이트 안의 페이지는 자유롭게 방문하되, 사이트 밖에서는 데이터를 가져오지 않는다. 유일한 예외는 같은 병원의 별도 도메인 사이트다(§1-1). 출력 스키마는 [`output_scheme.py`](reference/output_scheme.py)의 `HospitalHomepageResult`다.

병원 홈페이지는 구조가 제각각이고, 시술정보와 가격이 이미지나 동적 콘텐츠에 묻혀 있는 경우가 많다. 그래서 이 스킬에는 정해진 파싱 스크립트가 없다. 대신 에이전트가 Playwright MCP로 페이지를 직접 열어 탐색하고, 이미지에만 있는 정보는 vision으로 읽고, 카탈로그 매칭은 모델이 판단한다. 아래 §1~§8의 규칙은 실제 시행착오로 다듬은 결과다. 섹션 번호는 의존 순서다(수집 → 이미지 보강 → 매칭 → 정리) — 실제 작업은 페이지 단위로 수집·판독·매칭을 오가며 진행해도 된다. 규칙과 다르게 판단한 경우, 그 근거를 `crawl_metadata.notes`에 남긴다.

## 입력
- **`homepage_url`(필수)**: 수집할 공식 홈페이지 URL. 이 스킬은 URL을 찾아주지 않는다 — 받은 URL로만 동작한다.
- **병원 정보(선택)**: 병원DB의 아이디·병원이름·시도·시군구·읍면동·도로명주소. 아이디와 병원이름은 출력 파일명과 `hospital_id`/`hospital_name`에 쓰고, 전체 정보는 전달된 URL이 이 병원의 홈페이지가 맞는지 판단하는 데 쓴다.

## 출력
관계형 DB에 적재하기 좋은 형태로 정규화한 JSON 한 파일을 만든다.
- 스키마: [`output_scheme.py`](reference/output_scheme.py)의 `HospitalHomepageResult`
- 샘플: [`sample_output.json`](reference/sample_output.json)
- 저장형식: `output/{병원ID}_{병원이름}_homepage.json`

## 작업 예산과 멈춤 신호
비용·시간을 좌우하는 건 방문한 페이지 수가 아니라 **vision 판독 횟수**와 **순차 왕복(navigate→screenshot→Read) 수**다. 그래서 예산도 그 기준으로 관리한다.
- **텍스트 추출(`browser_evaluate`)은 사실상 무료다** — 페이지 수는 자유롭게 늘려도 된다. 텍스트로 거둘 수 있으면 vision을 쓰지 않는다.
- **vision 판독은 비싸다 — 사이트당 약 10장을 상한으로 본다**(가격 이미지는 §4 게이트 없이 이 한도 안에 포함). 풀페이지·초장축 캡처는 1장이 여러 장 값이니 §4 "효율 기법"으로 횟수를 줄인다.
- **멈춤 신호(에이전트가 관측 가능한 것):** ① vision 판독이 ~10장에 닿았다 ② 새 페이지가 이미 거둔 제품·장비만 반복한다(수확 체감) — 둘 중 하나면 새 페이지·이미지를 더 열지 말고 그때까지 모은 것으로 §7(체크리스트)~§8(마무리)로 간다.
- 멈출 때 못 끝낸 영역은 `crawl_metadata.follow_up`에 구조화해 남긴다(§7) — 후속 실행이 이어받는다. 완벽한 전수보다 "어디까지 했는지 기록된 부분 수집"이 낫다.
- 소요 시간은 10분 이내를 목표로 한다 — 실제 경과 시간이 늘어나는 주범은 순차 왕복(요청-응답 대기)이므로 §1(evaluate 우선)·§4(왕복 안티패턴) 기법으로 왕복을 줄이는 게 핵심이다.
- $ 단위 비용 상한(예: $2)은 에이전트가 실행 중 토큰 지출을 볼 수 없으므로 호출자·배치 러너가 바깥에서 관리한다. 호출자가 다른 예산을 주면 그것을 따른다.

## 0. 시작 — 접속·정찰·병원 확인
1. 주어진 `homepage_url`을 `browser_navigate`로 연다. 접속 자체가 안 되면(도메인 만료·차단·타임아웃) `crawl_metadata.errors`와 `notes`에 기록하고 멈춘다.
2. **정찰(한 번의 `browser_evaluate`)**: 홈에서 ① 전체 메뉴 링크(텍스트+URL) ② 푸터 사업자정보 ③ 본문 `innerText` 일부 ④ `<img>` 목록(alt·파일명)을 한 번에 뽑는다. 이 덤프 하나가 아래 병원 확인(3)과 전략 결정(4)의 공통 입력이다 — 무작정 캡처부터 하지 말고 여기서 사이트를 먼저 파악한다. 재사용 가능한 evaluate 스니펫은 [`snippets.md`](reference/snippets.md) 참고(복붙 출발점, 사이트에 맞게 조정).
3. **병원 확인**: 뽑아낸 병원명·주소 표기와 푸터·병원소개의 사업자정보(병원명·대표자·사업자등록번호·주소)를 입력받은 병원 정보와 비교해 `identity_status`(`match`/`mismatch`/`partial`)를 확정한다.
   - 명백히 다른 병원·업종(만료 도메인이 광고 사이트로 넘어간 경우 포함)이거나 `mismatch`: 수집하지 말고 근거를 `notes`에 남기고 멈춘다.
   - `partial`(예: 이름은 맞는데 주소가 다름 — 이전·분원 가능성): 계속 진행하되 `notes`에 남긴다.
   - 사업자정보를 못 찾았으면: 페이지 표기상 그 병원이 맞아 보이면 진행하고, 수집 중 확보되는 대로 확정한다.
   - 병원 정보를 입력받지 않았으면: 판정 불가 — `null`로 두고 사유를 `notes`에 남긴다.
4. **전략 결정**: 정찰 결과로 **초기** 수집 전략을 세운다 — 유형을 모르고 무작정 캡처부터 하면 느려진다. 분류 결과를 `crawl_method`에 한 줄로 적는다.
   - **텍스트형**(메뉴·본문·시술목록에 제품·장비명이 텍스트로 노출): vision 없이 텍스트·링크 추출 위주로 진행한다 — 가장 빠르다. 메뉴·시술목록에서 이름을 거의 다 거둘 수 있고, 상세 페이지는 모호한 항목 확인용으로만 들어간다.
   - **이미지형**(섹션 제목만 있고 내용·제품·가격이 이미지에 묻힘): §4 vision 게이트를 가동하되 §4의 "효율 기법"(파일명 먼저·개요→확대)을 반드시 지킨다.
   - **단, 메인 분류는 가설일 뿐이다.** 메인이 텍스트형이어도 상세·의료진·가격 페이지는 이미지인 경우가 흔하다 — 텍스트/이미지 판정은 **새 페이지에 도착할 때마다** 같은 값싼 `browser_evaluate`(본문 텍스트 길이 vs 이미지 수)로 다시 확인하고 vision 여부를 그때그때 정한다. 메인 정찰로 확정해도 되는 건 사이트 구조(원페이지·다중 페이지·다국어·프랜차이즈)와 신원뿐이다.
5. **출력 필드를 채운다**:
   - `homepage_url`: 실제 사용한 최종 URL(리다이렉트됐으면 도착한 URL)
   - `hospital_id`: 입력받은 아이디. 없으면 도메인으로 임시 ID를 만들고 `notes`에 "임시 ID 생성"이라 적는다
   - `hospital_name`: 입력받은 병원이름. 없으면 홈페이지(og:site_name·타이틀·로고)에서 가져온다

## 1. 방문 순서
홈페이지 구조는 제각각이지만, 정보 종류별로 아래 다섯 곳은 반드시 확인한다.

1. **홈**: 전체 메뉴를 파악한다. 다국어 스위처가 있으면 각 언어를 눌러 언어별 페이지 URL을 수집한다.
2. **병원 소개**: 운영정보(대표자·사업자번호·전화·영업시간·주차·교통·시설)와 의료진을 수집한다. §0에서 본 사업자정보는 재사용한다.
3. **시술·장비 목록**: 제품·장비명을 수집한다.
4. **시술 상세 페이지**: 목록에 걸린 상세 링크는 전부 들어가는 것이 기본이다. 병원이 마케팅용 자체 시술명을 짓는 경우가 많아, 실제 제품·장비명은 목록이 아니라 상세 본문(또는 본문 이미지)에만 드러나기 때문이다 — 실제로 목록만 보고 끝낸 사이트에서 제품의 95%를 놓친 적이 있다. 상세 본문을 끝까지 읽고, 방문한 URL을 `pages_visited`에 모두 남긴다. 시간·비용 때문에 전부 못 들어가면 방문 수와 미방문 수를 `notes`에 적는다 — 조용히 생략하지 않는다.
5. **가격·비급여 안내**: 시술가격을 확인한다. 이벤트 페이지에 가격이 없어도 끝내지 말 것 — 병원은 비급여 진료비를 고지할 법적 의무가 있어서, 성형외과·외과 계열도 "비급여진료비용 안내" 페이지에 일부 가격을 텍스트로 올려둔다. 푸터나 하단 링크에서 `비급여`·`nonpay`·`진료비용`을 찾아 반드시 방문한다.

각 페이지에서 탭·아코디언·"더보기"를 클릭해 숨은 콘텐츠까지 펼친다. 늦게 뜨는 콘텐츠는 `browser_wait_for`로 기다린다. 방문한 URL은 `crawl_metadata.pages_visited`에 누적하고 `pages_crawled`에 수를 센다.

**정찰(§0의 2단계)에서 쓴 `browser_evaluate` 텍스트 추출을 수집 내내 유지한다.** 페이지마다 `browser_snapshot`/스크린샷을 읽기보다 `innerText`·링크·이미지 목록을 evaluate로 뽑는 게 빠르고 정확하다 — 스냅샷은 클릭할 요소(ref)를 찾을 때만 쓴다. 메뉴·시술목록에 제품·장비명이 텍스트로 다 있으면 그것으로 §5 매칭을 끝내고, vision(§4)은 텍스트로 안 잡힐 때만 켠다.

### 1-1. 별도 도메인 사이트 — 같은 병원이면 수집한다
성형외과 등은 비수술 부문(필러·톡신·레이저)을 별도 도메인 사이트로 빼두는 경우가 꽤 있다. 거기는 이 프로젝트의 최우선 수집 대상(취급 제품·장비)이 몰려 있는 곳이고, 블로그·SNS와 달리 전담 추출 스킬도 없다 — 여기서 안 거두면 아무도 못 거둔다. 그래서 다음 조건으로 수집한다:
- 먼저 같은 병원인지 확인한다(§0의 2번과 같은 방법). 확인되면 그 사이트도 이 문서의 규칙대로 수집하고, 방문 URL을 `pages_visited`에, 경위를 `notes`에 남긴다.
- 주소가 다른 별개 지점(분원)이면 수집하지 않는다 — 병원DB에서 별개 병원이다. `branches.sibling_branches`에 기록만 한다.
- 한 단계만 따라간다. 그 사이트에서 또 다른 도메인이 이어지면 기록만 남긴다(비용 통제).

## 2. 추출
본 것을 원문 그대로 담는다 — 번역·요약·고쳐쓰기를 하지 않는다. 원문이 살아 있어야 카탈로그 매칭(§5)이 표기 속 단서(수식어·라인명)를 쓸 수 있고, 나중에 판단 근거를 검증할 수 있다. 특히 제품·장비·시술 이름은 사이트 표기 그대로 적는다 — 예를 들어 "울쎄라피 프라임"을 "울쎄라"로 줄여 적지 않는다. 정식 명칭으로 정리하는 건 §5(매칭)의 일이고, 전화번호·영업시간 같은 표기 통일은 §6(정규화)의 일이다.

## 3. 시술정보와 가격
시술은 `treatments[]`에 시술 단위로 담는다. 시술명은 사이트 표기 그대로 적고(§2), 시술에 쓰이는 제품·장비는 §5에서 매칭한 이름으로 `product_name`/`equipment_name`에 잇는다. 가격 필드의 의미와 형식(정가/판매가 구분, 범위 분해, 단위·수량, VAT)은 스키마 `TreatmentPrice`·`TreatmentPackage`의 필드 설명을 따른다. 아래는 가격을 확보하는 행동 규칙이다.

- 정가와 이벤트가가 함께 표기되면 둘 다 담는다 — 정가는 `regular_price`, 실제 판매가는 `low`/`high`. 마취비·사후관리비 같은 별도 비용 언급은 `notes`에 남긴다.
- **이미지로만 있는 가격은 vision으로 판독한다(§4)** — null로 두고 넘어가지 않는다. 이벤트 배너·프로모션 표·비급여 표가 이미지인 경우다. 가격에는 제품·장비 판독 같은 생략 게이트가 없다 — 이미지에 있으면 그게 유일한 출처다. 판독한 가격·시술·구성품은 텍스트 가격과 똑같이 담는다.
- 이벤트·시술 페이지에 가격이 없어도 끝내지 않는다 — 비급여진료비용 안내 페이지(§1의 5번)가 유일한 가격 출처인 경우가 많다. 텍스트든 이미지든 위 규칙대로 처리한다.

## 4. 이미지 판독(vision)
텍스트로는 없고 이미지에만 있는 정보를 캡처해 읽는다. 켜는 기준은 정보 종류에 따라 다르고(아래 두 버킷), 일단 켜면 "효율 기법" 순서로 비용을 줄인다. **어떤 경우든 그 정보가 텍스트로 있으면 텍스트로 읽고 vision은 쓰지 않는다** — §4는 "이미지에만 있는" 정보에만 해당한다.

**게이트 없음 — 이미지에만 있으면 그냥 판독한다.** 빠지면 안 되는 필수 정보라 켜고 끄기를 따지지 않는다.
- **가격**: 이벤트 배너·프로모션 표·비급여 표가 이미지면 그게 유일한 출처다. 무조건 판독한다(§3).
- **운영·신원·의료진 정보**: 진료시간표·주소/오시는길·주차안내·사업자정보·의료진 프로필이 이미지에만 있으면 판독한다 — 메뉴·본문은 텍스트인데 이 영역만 이미지인 사이트가 흔하다. 대개 항목당 1장이라 비용이 작다.

**게이트 있음 — 켜고 끄기를 판단한다(제품·장비).**
- **켠다**: 어떤 섹션에서 텍스트로 잡힌 제품·장비명이 0건인데 제품·장비로 보이는 이미지가 있으면, 그 섹션의 대표 이미지를 판독한다. 라벨만 보고 비우면 안 된다 — 실제로 제품·장비가 전부 이미지에 있던 사이트에서 라벨 매칭만 하다 100% 누락한 적이 있다.
- **얼마나 읽나**: 서로 다른 제품·장비로 보이는 이미지는 **각각 판독한다**(무누락 — 한 섹션에 제품이 여러 장이면 그만큼 읽는다. 실제로 스킨부스터 한 섹션에 제품 이미지가 8장이었던 적이 있다). 장식·중복(같은 제품을 여러 각도로) 이미지는 대표 1장이면 된다. 전체 vision은 사이트당 약 10장 예산(§작업 예산과 멈춤 신호) 안에서 쓰고, 아래 "효율 기법"(파일명·개요)으로 장당 비용을 줄인다. 텍스트 제품명이 충분히 잡히는 섹션에는 쓰지 않는다.
- **끈다**: 제품 0건이 이미지에 숨어서가 아니라 그 병원 진료영역상 원래 없는 것이면 돌리지 않는다. 코성형·양악 같은 수술 단일전문 의원은 주입제·장비를 원래 안 쓴다 — 진료 메뉴가 수술 일색이고 쁘띠·레이저·피부 섹션 자체가 없으면 0건이 정상이다. 이때는 `notes`에 "제품·장비 0건 — 수술 단일전문이라 미취급, vision 생략"을 남긴다.

**효율 기법 (vision은 비싸다 — 켜기로 정했어도 아래 순서로 줄인다)**
- **판독 전 파일명·alt·뷰어 제목부터 본다(vision 절약)**: 이미지의 `alt`, 파일명, 이미지 단독뷰의 제목이 곧 제품·장비명인 경우가 많다(예: `벨로테로.png`, `voluma_img.png`, `온다.png`, `비탈인젝터.png`). 자기식별되면 vision 없이 그 이름을 §5 매칭에 바로 쓴다. 텍스트로 잡힌 이름과 파일명이 일치하면 교차검증으로도 쓴다.
- **개요 먼저, 그다음 확대**: 제품·텍스트가 이미지 어디에 있는지 모를 땐, 먼저 작은 폭으로 축소한 1컷(또는 렌더되는 페이지면 전체 페이지 1컷)으로 위치만 파악하고, 글자가 있는 구간만 확대해 판독한다.
- **한 장씩 왕복하지 말 것(안티패턴)**: 제품 이미지가 여러 장이어도 개별 원본 URL로 `navigate→screenshot→Read`를 반복하는 것은 느리다. 렌더되는 페이지면 전체 페이지 한 컷으로 모으고, 안 되면 위 "개요→확대"로 캡처 횟수를 줄인다.
- **캡처와 판독**: 위 방법으로도 안 되는 이미지만 `browser_take_screenshot`으로 저장하고 `Read`로 읽는다. 판독한 값은 텍스트에서 얻은 값과 똑같이 담고 `verification_status`를 `verified`로 둔다.
- **추측 금지**: 흐릿하거나 잘려서 믿을 만하게 읽히지 않으면 그 항목만 비워 두고 `notes`에 "이미지 판독 불가"를 남긴다.
- **오버레이 우회**: 고정 오버레이(퀵메뉴·상담폼)가 이미지를 가리면, 이미지 원본 URL로 직접 이동하거나 오버레이를 숨기고 다시 캡처한다.
- **초장축 분할 캡처**: 아주 긴 세로 이미지는 통째로 읽지 않는다 — 다운스케일 때문에 글자가 뭉개진다. `browser_evaluate`로 그 `<img>`를 판독 가능한 고정폭으로 렌더한 뒤, 화면 높이만큼(약간 겹치게) 스크롤하며 구간별로 캡처해 순서대로 읽는다. (스니펫: [`snippets.md`](reference/snippets.md)의 3번)
- **교차검증**: 같은 내용이 텍스트로 있는 페이지가 있으면 대조한다.
- **기록**: 판독한 항목은 `notes`에 "○○ 이미지에서 vision 판독"이라고 출처를 남긴다.

## 5. 카탈로그 매칭
홈페이지에서 거둔(텍스트 + §4 vision으로 보강한) 제품·장비 이름을 카탈로그([`aesthetic_products.json`](reference/aesthetic_products.json)·[`aesthetic_equipments.json`](reference/aesthetic_equipments.json))와 대조해 취급 여부를 확정한다. 사이트 표기와 카탈로그 표기가 글자 그대로 일치하는 경우는 드물다 — 그대로 찾아보고 없다고 끝내지 말고, 아래 규칙을 모두 적용한다.

**후보를 빠짐없이 거둔다**
- 한 시술 설명에 여러 제품·브랜드가 나열되면 각각 추출한다. 시술명 하나로 뭉뚱그리지 않는다.
- 시술 설명 안에 장비명이 나오면 장비로도 수집하고, 그 시술의 `equipment_name`에 연결한다.
- 카탈로그에 없는 항목도 버리지 않는다 — `unmatched_products`/`unmatched_equipments`에 개별 항목으로 기록한다.

**이렇게 대조한다**
- 한↔영 교차: 사이트 표기와 카탈로그의 한글명·영문명을 양방향으로 대조한다. 예: "에코2"/"Eco 2" → eCO2.
- 유니코드 정규화(NFKC): CO₂↔CO2, 전각↔반각을 통일한 뒤 비교한다.
- 수식어·모델 넘버 제거: 떼어내고 브랜드로 대조한다. 예: "슈링크 유니버셜" → 슈링크.

**이렇게 판정한다**
- 보수적으로. 발음만 비슷하고 같은 제품이라는 확신이 없으면 매칭하지 않는다 — unmatched로 두고 `context`에 의심 근거를 남긴다.
- 브랜드만 있고 세부 라인이 불분명하면, 그 섹션의 카테고리에 맞는 대표 라인으로 매칭하고 `verification_status`를 `inferred`로 둔다.
- 홈·푸터의 제휴 제조사 로고·배너는 그 제조사와 거래가 있다는 신호다. 다만 제조사 하나에 제품이 여럿이라, 로고만으로 특정 제품을 확정하지는 않는다. 본문에서 그 제조사의 제품·브랜드가 확인되면 매칭의 보강 증거로 쓰고, 본문에 전혀 안 나오면 `manufacturer_signals`에 제조사 이름과 "제휴 배너 — 제품 미특정" 근거를 기록해 신호를 보존한다.

**unmatched로 보내기 전, 카탈로그를 한 번 더 조회한다(필수)**
"카탈로그 미수록"이라고 적기 직전에 브랜드명·영문명·NFKC·수식어 제거로 다시 찾아본다. 한글 시술명으로는 안 잡혀도 영문명·브랜드명으로는 있는 경우가 잦다 — 실제로 "리투오"(엘라비에 리투오), "버츄RF"(Virtue RF), "포텐자"(POTENZA)가 카탈로그에 있는데 미수록으로 잘못 보류된 적이 있다. `context`에 "미수록"이라 적을 땐 조회해서 없었다는 근거를 함께 남긴다 — 추측으로 적지 않는다.

## 6. 정규화
표기 형식과 필드 의미(전화번호·영업시간·주소 구조·`verification_status` 등)는 스키마의 필드 설명을 따른다. 여기에는 행동 규칙만 둔다.
- 여러 페이지에 나온 같은 제품·장비·의료진은 한 항목으로 합친다. 출처는 버리지 않는다 — 발견한 페이지마다 `sources`에 누적한다.
- `identity_status`는 §0에서 확정했다. 수집 중 확보한 사업자정보·운영정보가 그 판정과 어긋나면 갱신하고 근거를 `notes`에 남긴다. §0에서 사업자정보를 못 찾아 보류했다면 여기서 확정한다.
- 크롤링 중 발견한 다른 URL(언어 버전·구도메인)과 지점 정보는 `alternative_urls`·`branches`에 채운다.

## 7. 마무리 전 자가점검
최종 JSON을 내기 전, 아래 적신호를 점검한다. 해당하면 원인 영역을 다시 확인하고 결과를 `notes`에 남긴다.
- `operating_hours`가 비어 있다 → 접힌 토글·아코디언·탭 안에 있을 수 있다. 펼쳐서 재확인한다.
- 시술·장비 메뉴가 있는데 제품·장비가 0~1건이다 → 상세 미진입(§1의 4번) 또는 vision 미적용(§4) 의심. 다시 확인한다. 수술 단일전문이라 원래 없는 경우(§4의 "끈다")는 적신호가 아니다.
- `unmatched_*`만 있고 `matched_*`가 너무 적다 → 카탈로그 재조회(§5) 누락 의심.
- 장비·제품이 캐러셀·슬라이더에 있다 → 끝까지 넘겨 전부 거뒀는지 확인한다.

점검이 끝나면 `crawl_metadata`의 `completeness`·`notes`·`follow_up`을 채운다. 무엇을 담는지는 스키마의 필드 설명을 따른다. `notes`는 비워두지 않는다. **시간·예산·접근성 때문에 못 끝낸 영역(미방문 탭·판독 못 한 이미지·차단된 페이지 등)은 산문으로만 적지 말고 `follow_up`에 구조화한다** — 후속 실행이 이어받을 수 있게 `area`·`reason`·`detail`·`urls`를 채운다. 다 끝냈으면 빈 리스트로 둔다.

## 8. 마무리
- 산출 JSON을 `output/{병원ID}_{병원이름}_homepage.json`에 저장하고 검증한다:
  ```bash
  uv run python .claude/skills/hospital-homepage-extract/reference/output_scheme.py output/{병원ID}_{병원이름}_homepage.json
  ```
- `crawl_metadata.cost`는 실측값만 채운다 — 모델·소요 시간은 알면 적고, 토큰·비용은 모르면 null로 둔다. 추정해 채우지 않는다.
- 브라우저를 `browser_close`로 닫고, 캡처한 스크린샷 등 임시 파일(`.playwright-mcp/` 등)을 정리한다.

## Gotchas(주의사항)
실제 크롤링에서 막혔던 지점들이다 — 이 섹션이 가장 高신호다. 새 실패를 만나면 계속 추가한다.
- **홈페이지 구조·이미지 구성은 수시로 바뀐다.** 막히는 지점은 추측으로 채우지 말고 `notes`·`errors`에 남긴다.
- **이미지가 안 보인다(`0×0`/lazy/CSS배경/숨은 슬라이드).** 제품·가격 이미지가 lazy-load로 아직 `naturalWidth=0`이거나, `<img>`가 아니라 CSS `background-image`거나, 캐러셀의 숨은 슬라이드에 있을 수 있다. 스크롤로 로드를 유발하고, `<img>` src와 computed `background-image`를 둘 다 수집하고, 그래도 안 뜨면 이미지 원본 URL로 직접 열어 판독한다. (전체 페이지 스크린샷이 텅 비어 보이면 이 경우다.)
- **'비급여진료안내'가 가격표가 아닐 때가 많다.** 피부·미용 계열은 비급여 페이지·팝업이 진료비·제증명 수수료(진단서·소견서 등)만 고지하고, 실제 시술가는 이벤트/시술안내/인기시술에 있다. 비급여만 보고 "가격 없음"으로 끝내지 말 것(§3·§1의 5번).
- **팝업이 클릭으로 안 열리면 JS로 연다.** 고정 오버레이가 클릭을 가로채 `browser_click`이 실패하는 팝업(elementor 등)은 프레임워크 팝업 API(예: `elementorProFrontend.modules.popup.showPopup`)나 DOM 트리거로 연다.
- **프랜차이즈 신호.** 푸터에 지점 목록 + 가맹문의 + 사업자 2건이 보이면 프랜차이즈다. 이 도메인은 한 지점이고 나머지 지점은 주소가 다른 별개 병원 — 이 지점만 수집하고 형제 지점은 `branches.sibling_branches`에 이름만 남긴다(§1-1).
- **원페이지(앵커) 사이트.** 콘텐츠가 홈 한 페이지의 앵커 섹션에 다 있으면 `pages_crawled=1`, 출처 URL은 `…#anchor`로 적는다. 접힌 아코디언(진료시간 등)을 펼치는 것을 잊지 말 것.
- **`innerText`가 비어 나오는데 브라우저로 보면 내용이 있을 때(imweb 등 CSS 애니메이션 플랫폼).** `visibility:hidden` + 페이드인 애니메이션으로 감싼 위젯은 페이지 도착 직후 `innerText`를 뽑으면 텅 비거나 탭(`\t`)만 반환된다. 이 경우 `innerHTML`(소스)을 직접 검색하거나(`document.documentElement.innerHTML.search(/키워드/)`), `browser_wait_for(time=2)` 후 재시도한다 — 스크린샷이 비어 보여도 소스에 텍스트가 있을 수 있으니 vision으로 넘어가기 전에 HTML 소스를 먼저 확인한다.
